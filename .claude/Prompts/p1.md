────────────────────────────────────────
PROJECT PRINCIPLES
────────────────────────────────────────

1. Simplicity over complexity.

Whenever multiple approaches are possible, always recommend the simplest approach that satisfies the project's requirements.

Do not overengineer the solution.

This project should resemble a well-engineered open-source project, not an enterprise process-heavy application.

2. Repository-first documentation.

All project documentation will live inside the Git repository.

Do not propose Confluence, Notion, GitHub Wiki, or hybrid documentation.

Design the documentation architecture assuming everything is version-controlled inside the repository.

3. Avoid analysis paralysis.

The objective is to build a solid foundation quickly.

Do not create unnecessary planning artifacts.

If a document, diagram or process does not provide long-term value, it should not exist.

4. Practical documentation.

Documentation should explain architecture, decisions, development workflow, testing, and project direction.

Do not create documents for trivial decisions.

For example, UI color choices or small implementation details do not deserve dedicated documentation.

Architecture decisions, major design decisions, standards, and workflows do.

5. Version 0 remains planning only.

Do not produce implementation code.

Do not produce Kubernetes architecture.

Do not produce plugin architecture.

Do not produce RCA design.

Stay strictly inside Version 0.
