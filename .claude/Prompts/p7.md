We have now completed the Documentation Architecture Blueprint and want to establish the repository's permanent project documentation before continuing with Version 0.

Your task is to create the foundational repository documentation. Do not write implementation code.

Create the following:

1. Project Knowledge Base (the project's long-term memory)
2. Professional README.md
3. CONTRIBUTING.md
4. CODE_OF_CONDUCT.md
5. SECURITY.md
6. Decision Log
7. Contributor Reference

Requirements:

• The Project Knowledge Base becomes the canonical context document for all future contributors and AI assistants.
• It must capture the project's mission, North Star, philosophy, roadmap overview, current progress, completed goals, current goal, upcoming goal, accepted decisions, rejected ideas, glossary, contributor onboarding information, and AI instructions.
• It should be updated after every Goal completion (not after every task or phase). Only the progress-related sections should change; the mission and philosophy should remain stable.

README requirements:

• Professional open-source quality.
• Explain the project vision, motivation, architecture at a high level, roadmap, documentation structure, repository layout, contribution process, and future plugin ecosystem.
• Include multiple Mermaid diagrams where they improve understanding, including:
  - Vision
  - Documentation hierarchy
  - Architecture overview
  - Roadmap
  - Planned plugin ecosystem
• Keep diagrams readable and maintainable.

CONTRIBUTING requirements:

• Explain how contributors should get started.
• Explain the Project → Version → Goal → Phase → Task workflow.
• Explain branching strategy, pull requests, reviews, and documentation-first development.
• State that architecture and documentation must be approved before implementation.

Contributor Reference requirements:

• Summarize the current state of the project.
• Explain what has already been completed.
• Explain what is currently in progress.
• Explain how a new contributor should onboard and where to begin reading.
• This document should also be updated after every Goal completion.

Decision Log requirements:

• Maintain a concise chronological log of important architectural and product decisions.
• Each entry should include:
  - Date
  - Decision
  - Reason
  - Status

AI Rules:

• Future Claude sessions should begin by reading, in order:
  1. Project Knowledge Base
  2. Project Charter
  3. Documentation Architecture Blueprint
  4. Roadmap
  5. Current Goal documentation
  6. Active ADRs

General Rules:

• Keep documentation practical. Avoid enterprise-style bureaucracy.
• Prefer clarity over volume.
• Avoid duplicate information across documents by referencing the source of truth.
• Ensure consistency with the approved Documentation Architecture Blueprint.
• Do not modify the blueprint unless a conflict is discovered. If a conflict exists, stop and explain it rather than making assumptions.
